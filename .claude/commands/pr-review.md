# PR レビューコメント確認 & 対応

GitHub PR: **#$ARGUMENTS**

このワークフローを順に実行せよ。

---

## Step 1: PR レビューコメント取得

1. `gh pr view $ARGUMENTS --json reviews,comments,state,statusCheckRollup` で PR の状態を取得する
2. `gh api repos/{owner}/{repo}/pulls/{PR番号}/comments` でインラインレビューコメントを取得する（`gh pr view` は `reviewComments` フィールド非対応のため `gh api` を使う）
3. CI ステータスを確認する
4. Gemini を含む各レビュアーの未対応コメントと差し戻し有無を整理する

---

## Step 2: レビューコメントの分析

取得したレビューコメントを以下の基準で分類する。

### 分類基準

各コメントを **対象コードを実際に読んで** 以下に分類する:

- **正当な指摘**: 実際にバグ、セキュリティ脆弱性、仕様違反、パフォーマンス問題がある
- **的外れ**: コードを誤読している、存在しない問題を指摘している、プロジェクト規約と矛盾する
- **好みの問題**: どちらでもよいスタイルの提案（対応不要）

### 検証方法

1. 指摘されたコードを Read で実際に読む
2. 指摘の前提が正しいか検証する（POSIX 仕様、Node.js API 仕様、プロジェクト規約との整合性）
3. CLAUDE.md の Known Pitfalls と .claude/rules/ を参照し、プロジェクト固有の規約に照らす
4. 必要に応じて Context7 で公式ドキュメントを確認する
5. 判定時は `必須対応 / 推奨対応 / 任意 / 対応不要` も併記する

---

## Step 3: 分析結果の報告

ユーザーに以下の形式で報告する:

```
### レビューコメント分析

| # | レビュアー | 指摘内容（要約） | 判定 | 理由 |
|---|-----------|----------------|------|------|
| 1 | @reviewer | ... | 正当 / 的外れ / 好み | ... |

### 正当な指摘の詳細
（正当と判定したもののみ、修正方針を提示）
```

AskUserQuestion で対応方針を確認する:
- 正当な指摘を全て対応する
- 個別に選択する
- 対応不要（理由をコメントで返信）

---

## Step 4: 修正実施

ユーザーが承認した指摘に対して:

1. 修正を実施する
2. Quality Gate を通す:
   ```bash
   bun run test && bun run typecheck && bun run lint
   ```
   lint エラーがあれば修正内容を提示してから `bun run lint:fix` を実行する
3. 修正後、レビューゲートを再実行する
   - 常に `code-simplifier:code-simplifier`
   - 常に `.claude/skills/codex-review/SKILL.md` に従って `codex-review`
   - セキュリティ関連の変更または指摘がある場合は `.claude/rules/security.md` と CLAUDE.md Known Pitfalls に対する自己点検と回帰テストを再実行する
4. AskUserQuestion でコミットメッセージを確認してからコミット & push する（コミットメッセージに `#PR番号 レビュー対応:` を含める）
5. 必要に応じて PR コメントまたはレビュー返信で、採用した指摘と見送った指摘の理由を説明する

---

## Step 5: 学びの記録

今回のレビューで得た知見を評価する:

1. **CLAUDE.md Known Pitfalls への追記候補**: プロジェクト固有の落とし穴で、今後も再発しうるもの
2. **.claude/rules/ への追記候補**: コーディング規約やアーキテクチャルールの補強
3. **追記不要**: 一般的な知識、既に記載済み、プロジェクト固有でない

追記候補がある場合は AskUserQuestion でユーザーに内容を確認してから追記する。

---

## 注意事項

- レビュアーの指摘を鵜呑みにしない。必ず自分でコードと仕様を確認して判定する
- AI レビュアー（Gemini 等）は誤検出が多いため、特に慎重に検証する
- PR 上の指摘対応であっても、内部レビューゲートを省略しない
- セキュリティの妥当性は reviewer 任せにせず、ルール準拠とテストで確認する
- 的外れな指摘にも礼儀正しく対応する（PR コメントで理由を説明）
- 正当な指摘は素直に受け入れて修正する
